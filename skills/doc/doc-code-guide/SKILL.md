当用户提出“把运行命令记下来”“维护运行指南”“给模块补一个可执行命令示例”“更新 guide 文档”时，启用此 skill。

## 核心目标

将“可运行命令”沉淀为 `docs/guide/*.md`，并保证：

1. **一模块一文档**：每个模块/脚本单独一个 md
2. **可直接复制运行**：命令完整、参数明确
3. **可维护**：记录更新时间、适用范围、注意事项

## 固定目录与命名

- 根目录：`docs/guide/`
- 统一模板：`docs/guide/_template.md`
- 文档命名：`<module-or-script-name>.md`（例如 `token_prune_demo.md`）

## 维护流程

1. 先确认真实可运行命令（优先以当前项目实际脚本/CLI 为准）
2. 找到对应模块文档：
   - 已存在：增量更新
   - 不存在：基于 `_template.md` 新建
3. 必须更新文档中的：
   - `最后验证时间`
   - `适用代码版本/分支`（若无法确认，写“当前工作区”）
   - `注意事项`
4. 若运行方式发生变化：
   - 保留“变更记录”小节（简述旧→新）

## 文档最小结构（必须）

每个 `docs/guide/*.md` 至少包含：

1. 模块说明（这是什么）
2. 最小可运行命令（复制即跑）
3. 常见变体（可选参数）
4. 输出位置/结果检查方法
5. 最后验证时间
6. 注意事项

## 写作规范

- 命令用 `bash` 代码块
- 不写伪命令、不写“可能可以”
- 避免大段背景，聚焦“怎么跑”
- 若涉及环境依赖，必须写清楚（conda/env、模型路径、GPU 选择）

## 成功标准

一个合格的 `docs/guide/*.md` 应满足：

- 新成员能在不翻源码的情况下直接运行
- 命令具备可复制性和参数可解释性
- 文档可追踪更新时间与变更原因
