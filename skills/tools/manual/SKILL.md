# manual

项目的"运行手册"入口：`manual/`。

## 目录结构

```
manual/
├── bash/            # 每个模块一个 .sh，变量在顶部可编辑
│   └── <module>.sh
├── code/            # 每个模块一个 .md，最简 CLI 参数速查
│   └── <module>.md
└── guide/           # 按主题分类的精炼 notes，防项目幻觉
    └── <topic>.md
```

## 命名规则

- **顶层目录** 固定为 `manual/`（不用 `doc/`、`scripts/` 等别名）。
- **第二层子目录** 固定保留 `bash/`、`code/`、`guide/`。
- **文件名** 按**模块或主题**命名，用英文小写下划线，如 `placeobjects.sh`、`render.md`、`coordinates.md`。
  - 避免 `run_*.sh`、`e2e.md`、`main.md` 这种含义模糊的名字。
  - 一个文件只服务一个入口/一个主题，e2e 只是组合方式，不作为文件名。

## 何时更新

在以下任一情况下，主动更新 `manual/`：

1. 新增/修改/删除一个可运行模块（新脚本、新入口、新 CLI）。
2. 用户提到"更新文档"、"加个脚本"、"记个坑"、"怎么跑"。
3. 代码提交后同步刷新对应的 `bash/` 和 `code/` 条目。
4. 发现 LLM 容易误解的概念（如坐标系、依赖顺序、输出约定），写入 `guide/`。

## 写 bash/ 的规则

每个 `.sh` 只服务一个模块，变量暴露在文件顶部：

```bash
#!/usr/bin/env bash
set -euo pipefail

# ── 每次运行改这里 ──────────────────────────────────────────
PROMPT="<在此处直接编辑提示词>"
OUT_DIR="output/<模块目录>"
VIEWS=4   # 可选 1 | 2 | 4 | 6
BLENDER_PATH="/tmp/blender-4.0.2-linux-x64/blender"

# ── 下面不用改 ──────────────────────────────────────────────
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "$SCRIPT_DIR/../../../.." && pwd)"
mkdir -p "$OUT_DIR"
...
```

要点：
- 用户要改的变量集中在前 10 行，用分隔注释标出。
- `REPO_ROOT` 用相对定位从 `manual/bash/` 回到仓库根；如果模块层级不同，按实际层数调整 `../`。
- 只写"一次跑完"的最简路径，不堆积分支逻辑。
- 依赖外部二进制（如 Blender）时，路径以变量形式暴露在顶部。

## 写 code/ 的规则

每个 `.md` 只覆盖一个模块，格式：

```markdown
# <模块名>

## 最简运行

\```bash
python <路径> --prompt "<示例>" --output-dir <路径>
\```

## 参数说明

| 参数 | 必填 | 默认 | 说明 |
|------|------|------|------|
| --prompt | 是 | — | 场景描述 |
| --output-dir | 是 | — | 输出目录 |
| --views | 否 | 4 | 渲染视角数 (1/2/4/6) |

## 示例输出

- `<模块>/output/<示例>/lightgame_scene.json`
- `<模块>/output/<示例>/render_0.png`
```

要点：
- 只保留**实际会用到的参数**，砍掉 debug / 实验性 flag。
- 命令用 `bash` 代码块，不要用 `shell`。
- 附一个真实可跑的示例输出路径列表，让 LLM 知道跑完去哪找产物。

## 写 guide/ 的规则

每个 `.md` 只讲一个主题，格式：

```markdown
# <主题>

## 一句话结论

<用 1-2 句中文说清核心，防止幻觉>

## 关键坑点

- **<坑点标题>**：<为什么踩坑 + 正确做法>
- ...

## 相关文件

- `<路径>` — <一句话说明>
```

要点：
- **一句话结论**必须能直接复制给 LLM 作为"系统提示"片段，用来纠偏。
- **关键坑点**只写 LLM 容易误解的地方（坐标系、输出格式、依赖顺序、版本差异等）。
- 不写背景介绍、历史沿革、论文引用——只写"必须知道"。
- 文件命名与主题一一对应，如 `coordinates.md`、`coordinate-flow.md`、`render-views.md`。

## 更新流程

1. **定位**：确定改动涉及哪个模块，找到对应的 `manual/bash/<module>.sh` 和 `manual/code/<module>.md`。
2. **同步**：代码里的路径、参数、输出文件名变了，同步改 bash 和 code。
3. **补 guide**：如果改动引入新坑点（如坐标系变化、新增依赖），写入 `manual/guide/<topic>.md`。
4. **修剪**：检查是否有过时的条目（路径不存在、参数已废弃），直接删除或标记 `[已废弃]`。
5. **校验**：如果有 `bash/` 脚本，至少做一次 `bash -n <脚本>` 语法检查；code 里的路径用 `ls` 验证存在。

## 与其他 skill 的配合

- 代码改动后，通常先跑 `graphify update .` 同步知识图谱，再跑本 skill 更新 manual/。
- 如果用户问"怎么跑某个模块"，直接指向对应的 `manual/code/<module>.md`。
- 如果用户问"有什么坑"，直接指向 `manual/guide/<topic>.md`。
- 如果 `manual/` 缺失或命名不符合上述约定，自动整理并写入。
