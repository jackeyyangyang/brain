# 文档协作撰写工作流

这个 skill 提供一套结构化工作流，用于引导用户协作创建文档。作为主动引导者，带领用户走过三个阶段：**上下文收集**、**结构与打磨**、**读者测试**。

## 何时提供这个工作流

**触发条件：**
- 用户提到要写文档："写个 doc"、"起草一个 proposal"、"写个 spec"、"整理成文档"
- 用户提到特定文档类型："PRD"、"设计文档"、"决策文档"、"RFC"
- 用户看起来正准备开始一项较大的写作任务

**初始建议：**
向用户提供一个结构化的文档协作工作流，并解释三个阶段：

1. **上下文收集**：用户提供所有相关背景，Claude 负责追问澄清
2. **结构与打磨**：逐节迭代构建内容，头脑风暴、筛选、编辑
3. **读者测试**：用一个没有上下文的新 Claude 测试文档，提前发现盲区

解释这种方式能帮助确保文档在别人阅读时也能正常工作（包括别人把文档贴给 Claude 时）。询问用户是否愿意按这个流程来，还是更想自由发挥。

如果用户拒绝，就自由协作。如果用户接受，就进入阶段 1。

## 阶段 1：上下文收集

**目标：** 缩小“用户知道什么”和“Claude 知道什么”之间的差距，为后续的高质量引导打基础。

### 初始问题

先询问文档的元信息：

1. 这是什么类型的文档？（如技术规格说明、决策文档、proposal）
2. 主要读者是谁？
3. 希望读者读完之后产生什么效果？
4. 有需要遵循的模板或固定格式吗？
5. 还有其他约束或上下文需要知道吗？

告诉用户可以用简写回答，也可以直接一股脑倾倒信息。

**如果用户提供了模板或提到某种文档类型：**
- 问用户是否有模板文档可以分享
- 如果给了共享文档链接，使用合适的集成读取
- 如果给了文件，就读取文件

**如果用户提到要编辑现有共享文档：**
- 使用合适的集成读取当前文档状态
- 检查是否存在没有 alt-text 的图片
- 如果有图片缺少 alt-text，解释：当别人用 Claude 理解文档时，Claude 无法“看到”这些图片。询问是否需要为图片生成 alt-text；如果需要，请用户把图片贴到聊天里以便生成描述性 alt-text

### 信息倾倒

当初始问题回答后，鼓励用户把已知上下文全部倒出来。可以要求的信息包括：
- 项目 / 问题背景
- 相关团队讨论或共享文档
- 为什么不采用其他方案
- 组织背景（团队关系、历史事件、政治因素）
- 时间压力或限制
- 技术架构或依赖
- 利益相关方担忧

提醒用户先不要管组织结构，先把信息全倒出来。可以提供多种方式：
- 流水账式 info dump
- 指向团队频道或讨论串让你去读
- 提供共享文档链接

**如果有可用的集成**（如 Slack、Teams、Google Drive、SharePoint 或其他 MCP server），可以告诉用户这些可以直接拉取上下文。

**如果没有集成且当前在 Claude.ai 或 Claude app：** 建议他们在 Claude 设置里启用 connectors，这样就能直接从消息应用和文档存储中提取上下文。

告诉用户：等他们完成第一轮信息倾倒后，你会继续追问澄清。

### 收集过程中的行为规则

- 如果用户提到团队频道或共享文档：
  - 如果有集成：告知即将读取，然后读取
  - 如果没有集成：说明当前无法访问，建议启用 connectors 或直接把内容贴出来

- 如果用户提到未知实体/项目：
  - 问用户是否要搜索已连接工具以了解更多
  - 在得到确认前不要搜索

- 在用户提供上下文的过程中，持续跟踪：已经理解了什么、还有哪些点不清楚

### 提问澄清问题

当用户表示第一轮信息已经说完（或已经给出了大量上下文）时，提出澄清问题以确保理解充分：

生成 **5-10 个编号问题**，针对当前理解中的空缺。

告诉用户可以用简写回答（比如：`1: 是，2: 看 #channel，3: 不行因为要兼容旧版本`），也可以继续贴文档、指频道、继续 info dump，什么高效用什么。

**退出条件：**
当你的问题已经体现出对背景的理解，开始能够直接问边界情况和权衡，而不再需要基础背景解释时，说明上下文已经收集得足够。

**过渡：**
问用户：现在还有没有更多背景想补充，还是该进入起草阶段了？

如果用户还要补充，就继续。准备好后进入阶段 2。

## 阶段 2：结构与打磨

**目标：** 按 section 逐步构建文档，通过头脑风暴、筛选和迭代编辑完成内容。

### 给用户的说明

解释接下来会按 section 来构建文档。对每个 section：
1. 先问澄清问题，搞清楚这个 section 应该包含什么
2. 头脑风暴 5-20 个可写入的点
3. 用户决定哪些保留 / 删除 / 合并
4. 起草该 section
5. 通过局部编辑继续精修

从**未知最多的 section**开始，通常是核心决策 / 核心方案。剩余部分再逐个推进。

### Section 顺序

**如果文档结构已经明确：**
- 问用户想先从哪个 section 开始
- 建议优先从未知最多的 section 开始
  - 对决策文档来说，通常是核心 proposal
  - 对技术 spec 来说，通常是技术方案部分
  - 摘要类 section 最好留到最后

**如果用户不知道应该有哪些 section：**
- 根据文档类型和模板，建议 3-5 个合适的 section
- 询问这个结构是否合适，是否需要调整

### 一旦结构确定

创建初始文档骨架，为所有 section 放占位内容。

**如果能使用 artifacts：**
- 用 `create_file` 创建 artifact
- 告诉用户即将创建一个带有所有 section 占位符的初始结构
- 创建 artifact，包含所有标题和简短占位语，如 `[待补充]` 或 `[此处填写内容]`
- 给出 scaffold 链接，并进入逐节填充

**如果不能使用 artifacts：**
- 在工作目录创建一个 markdown 文件，例如 `decision-doc.md`、`technical-spec.md`
- 告诉用户即将创建初始骨架
- 写入所有标题和占位内容
- 确认文件名，并进入逐节填充

## 对每个 section 的工作流

### 步骤 1：澄清问题

明确宣布：现在开始处理 `[SECTION NAME]`。

针对该 section 的目标和上下文，提出 **5-10 个具体问题**，确认该 section 应写什么。

告诉用户可以用简写回答，也可以只指出哪些点最重要。

### 步骤 2：头脑风暴

针对 `[SECTION NAME]`，列出 **5-20 个可写入点**（数量取决于复杂度），重点考虑：
- 用户已经提过、但可能会遗漏的背景
- 还没明确提到、但值得纳入的角度或考量

用编号列出这些选项。结尾可以补一句：如果还想扩展，可以继续 brainstorm。

### 步骤 3：筛选

让用户指出：
- 哪些保留
- 哪些删除
- 哪些合并

并尽量让用户给出简短理由，以便你学习他们在后续 sections 中的偏好。

可以给一些格式例子：
- `保留 1,4,7,9`
- `删掉 3（和 1 重复）`
- `删掉 6（读者已经知道）`
- `合并 11 和 12`

**如果用户给的是自由反馈**（如“整体不错，不过……”），也要能抽取偏好并继续，不要求必须严格按编号选择。

### 步骤 4：查漏补缺

根据用户选中的内容，再问一句：这个 section 还有没有重要但没覆盖的内容？

### 步骤 5：起草

使用 `str_replace` 把该 section 的占位内容替换成真正的草稿。

明确告知用户：现在会根据他们的选择起草 `[SECTION NAME]`。

**如果使用 artifacts：**
- 起草后给出 artifact 链接
- 请用户阅读并指出想改的地方
- 强调具体反馈更有助于后续 section 对齐风格

**如果使用文件：**
- 起草后确认该 section 已写入 `[filename]`
- 请用户阅读并指出要修改的地方
- 同样提醒：越具体越有助于后续 section

**第一次起草 section 时要补充的关键说明：**
最好不要直接自己改文档，而是告诉你想怎么改。这样有助于你学习他们的写作风格和偏好。比如：
- “删掉 X 这个 bullet，已经被 Y 覆盖了”
- “把第三段写得更简洁一点”

### 步骤 6：迭代精修

当用户给出反馈时：
- 用 `str_replace` 局部修改
- **如果用 artifact**：每次修改后重新给链接
- **如果用文件**：只需确认已完成修改
- 如果用户自己直接改了文档并让你看，要记住他们怎么改的，把这些偏好带到后续 sections 中

持续迭代，直到用户对这个 section 满意。

### 质量检查

如果已经连续 3 次修改都没有实质变化，可以问一句：
- 有没有什么内容可以删掉但不损失信息量？

当这个 section 完成后，明确确认 `[SECTION NAME]` 已完成，并问是否进入下一个 section。

对所有 section 重复上述流程。

## 接近完成时

当 80% 以上 sections 已完成时，明确告诉用户：你将重新通读整篇文档，并检查：
- 各 sections 之间是否连贯一致
- 是否有重复或矛盾
- 是否存在空泛、模板化、“slop”式内容
- 是否每一句话都真正有信息量

阅读全文并给出反馈。

**当所有 sections 都写完并精修过后：**
- 明确说明文档已全部起草完成
- 再做一次完整 review
- 检查整体连贯性、完整性和流畅度
- 给出最终建议
- 问用户是否进入“读者测试”，还是还想再微调

## 阶段 3：读者测试

**目标：** 用一个没有上下文污染的“新 Claude”测试文档，验证文档对真实读者是否有效。

### 给用户的说明

告诉用户：接下来要测试的是，这份文档对读者来说是否真的可用。这样能提前发现盲区——那些作者自己觉得理所当然，但别人可能会困惑的地方。

## 测试方式

### 如果可以使用子代理（如 Claude Code）

直接执行测试，不需要用户手动参与。

#### 步骤 1：预测读者问题

说明：先预测读者在尝试理解或发现这篇文档时，会问哪些问题。

生成 **5-10 个现实读者会问的问题**。

#### 步骤 2：用子代理测试

说明：这些问题会拿给一个没有当前对话上下文的新 Claude 来回答。

对每个问题：
- 只把文档内容和问题传给子代理
- 观察 Reader Claude 回答得对不对、哪里模糊、哪里误解
- 汇总它答对了什么、答错了什么

#### 步骤 3：额外检查

说明：还会额外检查：
- 歧义
- 错误假设
- 内部矛盾

调用子代理执行这些检查，并总结问题。

#### 步骤 4：汇报并修复

如果发现问题：
- 明确说明 Reader Claude 在哪些地方卡住了
- 列出具体问题
- 说明接下来要修补这些缺口
- 回到相应 section 做 refinement

---

### 如果不能使用子代理（如 Claude.ai 网页版）

这时需要用户手动完成测试。

#### 步骤 1：预测读者问题

问用户：别人为了理解这篇文档，会在 Claude.ai 里输入什么问题？

然后生成 **5-10 个现实读者会问的问题**。

#### 步骤 2：设置测试方式

给出操作说明：
1. 打开一个新的 Claude 会话：https://claude.ai
2. 粘贴或分享文档内容（如果是共享文档平台且已启用 connectors，就直接给链接）
3. 把生成的问题逐个问给 Reader Claude

对于每个问题，让 Reader Claude 同时给出：
- 它的答案
- 哪些地方模糊或不清楚
- 文档默认读者已经知道哪些背景

检查 Reader Claude 是否回答正确，是否误解了内容。

#### 步骤 3：额外检查

再让 Reader Claude 回答这些问题：
- “这份文档中有哪些地方对读者来说可能模糊或不清楚？”
- “这份文档默认读者已经知道哪些背景？”
- “文档里是否存在内部矛盾或不一致？”

#### 步骤 4：根据结果继续迭代

询问用户：Reader Claude 哪些地方答错了、误解了、卡住了？

然后回到相应 section 修补这些缺口。

---

### 两种方式的退出条件

当 Reader Claude 能稳定正确回答问题，并且不再暴露新的模糊点或缺口时，就说明文档已经准备好了。

## 最终审阅

当读者测试通过后：

1. 建议用户自己再通读一遍——这是他们的文档，他们对其质量负责
2. 建议再次核对事实、链接、技术细节
3. 让用户确认这份文档是否实现了最初设定的影响目标

然后问用户：
- 还要不要再做最后一轮 review？
- 还是可以收工了？

**如果用户还要最后 review，就继续。否则：**
说明文档已完成，并给几点收尾建议：
- 可以考虑在 appendix 里放上这段协作对话，方便读者理解文档是如何形成的
- 可以用 appendix 提供更深的补充，而不让正文膨胀
- 当真实读者给出反馈后，要及时回写文档

## 有效引导的提示

### 语气
- 直接、程序化
- 只有在影响用户行为时，才简要解释原因
- 不要“推销”这个流程，直接执行即可

### 处理偏离流程的情况
- 如果用户想跳过某个阶段：直接问是否要跳过并改为自由协作
- 如果用户显得烦躁：承认这比预期更花时间，并提出更快推进的方法
- 永远给用户保留调整流程的主动权

### 上下文管理
- 在整个过程中，只要发现某个被提到的点上下文不足，就要主动补问
- 不要让信息缺口一直积累

### Artifact 管理
- 用 `create_file` 起草完整 section
- 用 `str_replace` 执行所有编辑
- 每次修改后都给 artifact 链接
- 不要把 artifact 用于 brainstorm 列表——那部分应留在对话里

### 质量优先于速度
- 不要急着走完流程
- 每一轮迭代都应该产生实际改进
- 目标是得到一份**对读者真的有效**的文档
