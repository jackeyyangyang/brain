当用户完成代码修改、要求提交代码、要求整理提交信息、或要求写日报时，启用此 skill。

## 目标

输出一套**可直接复制**的收尾流程，分为两部分：

1. **Commit 指导**：先自动执行 `git status` 展示当前状态，再根据改动内容整理 **1 到 3 个** `git add` / `git commit -m` 建议，每个 commit 都要附一个规范、简短的 commit message。
2. **日报追加**：根据**当天日期**生成可追加到当日日报的详细记录内容；如果拆成多个 commit，就为每个 commit 分别生成一段，包含真实 commit 编号、时间、名称和较详细的修改说明。

## 强约束

- **自动执行** `git status`，用于先展示当前工作区状态。
- **不要自动执行** `git add`、`git commit`、`git push`。
- **不要擅自提交**代码。
- **不要使用 `--trailer` 参数**，例如 `git commit -m "..." --trailer "Co-authored-By: ..."`。如用户主动要求方可添加。
- 默认只提供：
  - 自动执行后的 `git status` 结果
  - 1 到 3 组建议执行的命令
  - 对应的 commit message
  - 对应的日报追加文本
- 如果没有足够上下文判断改动内容，先基于当前任务给出**最保守、最概括**的描述，不要编造细节。
- commit message 必须简洁，优先中文。

## Commit message 规范

格式：

```text
<type>: <简短描述>
```

可用类型：

- `feat`：新功能
- `fix`：修复问题
- `docs`：文档改动
- `refactor`：重构
- `chore`：杂项维护
- `ci`：CI/CD 或脚本流程
- `test`：测试相关

要求：

- 标题 **≤ 72 字符**
- 使用**祈使语气**
- **中文优先**
- **不带句号**
- 聚焦“本次改动的核心价值”，不要把很多动作堆在一条 message 里

## Commit 拆分规则

根据 `git status` 的结果判断是否应该拆分多个 commit。

- 默认输出 **1 个** commit
- 如果改动明显属于不同主题，可拆成 **2 个或 3 个** commit
- **最多只给 3 个** commit，避免过度切碎
- 拆分依据应优先看：
  - 不同功能模块
  - 不同改动类型（代码 / 文档 / 配置）
  - 是否能形成独立可回顾的提交单元
- 如果改动虽然多，但本质上服务同一个目标，仍然应合并成 1 个 commit
- 每个 commit 都必须分别给出：
  - 该 commit 的真实 commit 编号（如果尚未实际提交，则明确标注为“待生成 commit hash”）
  - 建议暂存命令
  - 建议的 commit message
  - 建议的提交命令
  - 对应的日报追加内容
- 如果判断依据不足，不要强拆；优先给出更保守的 1 个 commit 建议

## 输出顺序

严格按以下顺序输出：

### 1. Commit 建议

先自动执行 `git status` 并展示结果。

然后给一句总结，例如：

- `建议按 2 个 commit 拆分，再补日报。`
- `建议本次合并为 1 个 commit，再补日报。`

然后按顺序给出 `Commit 1` 到 `Commit 3` 的建议；如果不需要拆分，就只给 `Commit 1`。其中“提交编号”字段应填写真实 commit hash；若尚未提交，则写 `待生成 commit hash`。

#### A. 状态检查结果

自动执行：

```bash
git status
```

并展示结果。

#### B. 每个 commit 的建议内容

对每个 commit，严格按下面结构输出：

##### Commit N

###### 1) 建议暂存改动

如果知道具体文件，就给精确命令：

```bash
git add path/to/file1 path/to/file2
```

如果不知道具体文件，就给保守版本：

```bash
git add .
```

###### 2) 建议 commit message

单独给出一行：

```text
fix: 修复xxx问题
```

###### 3) 建议提交命令

```bash
git commit -m "fix: 修复xxx问题"
```

### 2. 日报追加内容

根据当天日期，生成**可直接追加**到日报文件中的文本。

如果拆成多个 commit，就为每个 commit 分别生成一段。

默认格式如下：

```markdown
## Commit <真实 commit hash | 待生成 commit hash> | HH:MM | fix: 修复xxx问题

- **提交编号**: `<真实 commit hash | 待生成 commit hash>`
- **提交时间**: `HH:MM`
- **提交名称**: `fix: 修复xxx问题`
- **本次修改概述**: 用 1 句话说明这次提交解决了什么问题或完成了什么目标
- **详细修改内容**:
  - 修改了哪些文件或模块
  - 每个文件大致改了什么
  - 关键逻辑、字段、命令或行为发生了什么变化
- **修改原因**: 说明这次为什么要改，背景问题是什么
- **验证结果**: 说明已做的验证；如果未验证，明确写“未执行验证”
- **影响范围**: 说明会影响到哪些模块、脚本、命令、输出或使用方式；如果没有，写“无”
- **后续事项**: 下一步计划；如果没有，写“无”
```

## 日期与文件规则

- 日报文件名默认使用当天日期：`daily_logs/YYYY-MM-DD.md`
- 如果用户只说“帮我写日报追加内容”，默认按**当天日期**生成。
- 如果用户项目里已有不同日报格式，优先**贴合现有格式**。
- 如果你看不到现有日报内容，也不要编造历史内容；只输出这次要追加的片段。

## 内容生成规则

### 提交编号 / 时间 / 名称

每个日报片段都必须先写清楚：

- 提交编号：填写真实 commit hash；如果尚未实际提交，写 `待生成 commit hash`
- 提交时间：使用生成该建议时的时间，如 `15:30`
- 提交名称：直接使用对应 commit message

### 本次修改概述

用 1 句话概括本次提交最重要的结果，例如：

- `补充 Cursor skill，使其支持按改动主题拆分多个 commit。`
- `修复场景生成配置解析异常，避免运行时报错。`

### 详细修改内容

这一部分要写得足够让用户后续 review 时能快速想起改了什么，至少应覆盖：

- 改了哪些文件、目录或模块
- 每个主要文件分别改了什么
- 有哪些关键逻辑、参数、字段、命令、输出结构或行为变化
- 如果是文档或配置改动，要说明新增了什么规则、约束或使用方式

如果上下文足够，优先按“文件 / 模块 → 具体变化”的方式展开，不要只写空泛总结。

### 修改原因

写清楚这次为什么改，背景问题是什么。例如：

- 旧流程只能输出单个 commit 建议，不利于按主题整理提交
- 日报模板过于简略，后续回顾时难以快速恢复上下文

### 验证结果

优先写客观结果：

- `已检查相关文件内容与输出格式。`
- `已完成本地运行验证。`
- `未执行验证。`

### 影响范围

说明会影响到哪些模块、脚本、命令、输出、协作流程或使用方式；如果没有，写：

- `无`

### 后续事项

保持一句话：

- `等待用户执行对应 commit 并追加到当日日报。`
- `后续可按实际改动补充更细的验证记录。`
- `无`

## 响应风格

- 全程使用中文。
- 输出要短、稳、可复制。
- 不写大段铺垫。
- 用清晰分段：`Commit 建议` / `日报追加内容`

## 决策优先级

当出现冲突时，按以下优先级处理：

1. 用户当前明确要求
2. 项目内已有 commit / 日报规范
3. 本 skill 默认模板

## 成功标准

当用户在一次代码修改后请求“帮我提交建议”或“顺便写日报”时，你的响应必须满足：

- 给出 **1 到 3 条**规范的 commit message
- 给出 **每个 commit 可复制**的 git 命令
- 给出 **每个 commit 可直接追加**的日报片段
- 日报内容必须包含 **真实 commit 编号、时间、名称、详细修改内容**
- **仅自动执行** `git status`，**不自动执行**其他 git 命令
