当用户想理解代码库结构、模块职责、调用链、数据流、概念关系，或要求“先用 graphify 看看”时，启用此 skill。

## 核心原则

**先结构，后细节；先 graphify，后源码。**

不要一开始就全局搜索或大量读文件。先用 graphify 找到关键节点、关键路径、关键文件，再做定点阅读。

## 适用问题

- `这个模块是干什么的？`
- `主流程怎么走？`
- `A 和 B 什么关系？`
- `数据怎么从 X 到 Y？`
- `render/benchmark/evaluator 各自在哪里串起来？`
- `这个 repo 大致怎么组织？`

## 默认工作流

### 1. 宽问题：先 `query`

适合：

- 架构理解
- 主流程理解
- 模块职责理解

命令：

```bash
graphify query "<用户原问题>"
```

目标：

- 找主链路
- 找关键节点
- 缩小源码阅读范围

### 2. 双点关系：先 `path`

适合：

- A 和 B 的关系
- 数据/控制流路径
- 配置如何作用到运行逻辑

命令：

```bash
graphify path "A" "B"
```

如果路径不够清楚：

- 对中间关键节点执行 `graphify explain "<node>"`
- 再按需读 1～3 个源码文件

### 3. 单点解释：先 `explain`

适合：

- `X 是什么？`
- `这个节点为什么重要？`
- `这个类/函数大概处于什么上下文？`

命令：

```bash
graphify explain "X"
```

### 4. 整体项目导航

优先顺序：

1. 看 `graphify-out/wiki/index.md`（如果存在）
2. 再跑 `graphify query "<architecture question>"`
3. 不够再读 `graphify-out/GRAPH_REPORT.md`
4. 最后才广泛读源码

## 回答输出结构

优先按两层输出：

### 第一层：结构结论

说明：

- graphify 找到了哪些关键节点
- 主链路或关键路径是什么
- 当前最可信的结构性结论是什么

### 第二层：源码确认

只有在需要时补充：

- 我进一步看了哪些文件
- 哪些实现细节只能从源码确认

## 源码阅读约束

graphify 之后，只读最相关的少量文件：

- 优先 1～3 个文件
- 优先读入口、协调层、核心类
- 避免一开始就扫整片目录

## 何时不能只靠 graphify

以下情况必须回源码：

- 需要逐步解释具体实现
- 需要确认参数、返回值、条件分支
- 需要排查 bug 或异常路径
- 需要写代码

## 质量标准

好的回答应该：

- **先给主结构，再给实现细节**
- **能指出关键节点与关键路径**
- **减少无关文件阅读**
- **让用户知道接下来该看哪几个文件**

## 修改后要求

如果本次会话修改了代码，结束前执行：

```bash
graphify update .
```
