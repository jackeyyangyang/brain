当用户准备改代码，或问“改这里会影响什么”“谁依赖它”“先帮我评估风险”时，启用此 skill。

## 核心原则

**改代码前先评估影响面。**

先用 graphify 找影响范围、关键依赖、潜在传播路径，再决定读哪些源码、改哪些文件、跑哪些验证。

## 适用问题

- `改这个会影响哪里？`
- `谁依赖这个类/函数？`
- `删掉这层抽象会波及什么？`
- `我准备重构这个模块，风险点在哪？`
- `这个 prompt builder 改动后哪些流程要复测？`

## 默认工作流

### 1. 影响面初筛：先 `affected`

命令：

```bash
graphify affected "X"
```

目标：

- 找直接依赖
- 找间接受影响节点
- 评估改动扩散半径

如需更深层分析：

```bash
graphify affected "X" --depth 3
```

如需限定关系：

```bash
graphify affected "X" --relation <relation>
```

### 2. 关键路径核实：补 `path`

如果用户关心“改动怎么传到另一处”：

```bash
graphify path "X" "Y"
```

适合：

- 从改动点到用户可见行为的路径
- 从底层工具到上层流程的传播路径

### 3. 关键节点解释：补 `explain`

如果某个受影响节点不清楚：

```bash
graphify explain "<node>"
```

目标：

- 判断它是否真是高风险点
- 判断它在流程里是核心还是边缘

### 4. 再做源码确认

graphify 之后，再读最值得关注的源码：

- 改动点本身
- 直接依赖方
- 关键调用链中最核心的 1～3 个文件

## 输出要求

回答尽量包含 4 部分：

### 1. 影响范围

- 直接影响哪些节点
- 间接影响哪些流程

### 2. 风险等级

粗分即可：

- 低：局部工具、边缘节点
- 中：一条主链路上的中间节点
- 高：入口、协调层、公共基础组件

### 3. 建议关注文件

给出最值得读/改/复查的文件或模块。

### 4. 验证建议

根据影响面建议：

- 哪类测试最值得先跑
- 哪条主流程最值得先手动验证
- 哪些上游/下游场景容易被回归影响

## 何时不能只靠 graphify

以下情况必须进一步看源码：

- 要确认实际分支逻辑
- 要看兼容层、异常处理、默认值
- 要做精确修改
- 要确定测试入口

## 改后要求

完成修改后必须执行：

```bash
graphify update .
```

目标：

- 同步新图谱
- 避免下一轮分析基于旧关系

## 可选知识沉淀

如果这次影响分析结论质量很高，可考虑保存：

```bash
graphify save-result --question "..." --answer "..." --type query --nodes "..."
```

## 质量标准

好的影响分析应该：

- **明确直接影响与间接影响**
- **指出高风险节点或路径**
- **给出建议验证范围**
- **帮助用户决定是否值得改、该先改哪**
