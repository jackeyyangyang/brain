当用户要理解代码库、分析模块关系、定位调用链、评估改动影响、复盘知识、或要求“尽量用 graphify”时，启用此 skill。

## 核心原则

**先 graphify，后源码。**

默认不要一上来就大量读文件或全局搜索。先使用 graphify 把问题压缩成更小的上下文，再按需读取少量源码。

## 目标

把 graphify 用到最有价值的 5 类任务上：

1. **架构理解**：快速回答“这个模块是干什么的、跟谁有关”
2. **关系追踪**：找两个概念、模块、函数之间的路径
3. **定点解释**：解释某个符号、文件、类、函数的上下文
4. **影响分析**：改某处代码会影响哪些节点
5. **知识沉淀**：把一次问答保存到 graphify memory，形成复用知识

## 默认工作流

### 场景 1：用户问“这块代码怎么工作”

优先顺序：

1. 运行 `graphify query "<用户问题>"`
2. 如果问题聚焦某个概念，再运行 `graphify explain "<concept>"`
3. 只有 graphify 输出还不够时，才读取相关源码

适用示例：

- `这个场景生成流程怎么工作的？`
- `benchmark 的执行链路是什么？`
- `render 相关逻辑都在哪？`

### 场景 2：用户问“X 和 Y 什么关系”

优先运行：

```bash
graphify path "X" "Y"
```

如果路径不够清楚：

- 再对路径中的关键节点运行 `graphify explain "<node>"`
- 再按需读取少量源码

适用示例：

- `RenderManager 和 benchmark runner 什么关系？`
- `配置文件是怎么影响渲染流程的？`

### 场景 3：用户问“改这里会影响什么”

优先运行：

```bash
graphify affected "X"
```

如果需要更具体：

- 可补 `--depth 3`
- 可补 `--relation <relation>`

适用示例：

- `如果我改这个 evaluator，会影响哪些流程？`
- `这个 prompt builder 改动后波及面有多大？`

### 场景 4：用户要求整体理解项目

优先顺序：

1. 看 `graphify-out/wiki/index.md`（如果存在）
2. 不够再用 `graphify query "<architecture question>"`
3. 仍不够再读 `graphify-out/GRAPH_REPORT.md`
4. 最后才大范围读源码

### 场景 5：完成修改后

如果本次会话里修改了代码文件，**仅在需要同步图谱时**执行：

```bash
graphify update .
```

建议触发时机：

- 本次确实改了代码文件
- 用户明确要求“顺手更新 graphify”
- 会话结束前做一次集中更新，而不是每次小改都更新

目标：保持图谱与当前代码同步，但**不需要高频刷新**。

## 提问到命令的映射

### 1. 宽问题

如果用户问：

- `这个模块是干什么的？`
- `这里的主流程是什么？`
- `这个 repo 怎么组织的？`

优先用：

```bash
graphify query "<原问题>"
```

### 2. 双点关系问题

如果用户问：

- `A 和 B 有什么关系？`
- `数据怎么从 X 流到 Y？`

优先用：

```bash
graphify path "A" "B"
```

### 3. 单点解释问题

如果用户问：

- `X 是什么？`
- `为什么这里会有这个节点？`

优先用：

```bash
graphify explain "X"
```

### 4. 改动影响问题

如果用户问：

- `改这个会影响哪里？`
- `谁依赖它？`

优先用：

```bash
graphify affected "X"
```

### 5. 问答沉淀

当一次 graphify 问答已经得到比较稳定、可复用的答案时，建议保存：

```bash
graphify save-result --question "..." --answer "..." --type query --nodes "..."
```

注意：

- 只在答案质量较高、可复用时保存
- 不要把明显不完整、猜测性强的回答写入 memory

## 何时不该只靠 graphify

以下情况，graphify 只能作为第一步，不能代替源码阅读：

- 需要修改具体实现
- 需要确认函数参数、返回值、边界条件
- 需要看真实异常处理和分支逻辑
- 需要核对最新代码是否已偏离图谱

这时应采用：

1. 先 graphify 缩小范围
2. 再读取最相关的 1～3 个文件
3. 必要时再做精确搜索

## 响应要求

输出应尽量分为两层：

### 第一层：graphify 结论

先告诉用户：

- graphify 看到了什么
- 哪些节点/路径最关键
- 当前最可信的结构性结论是什么

### 第二层：源码确认

如果需要，再补：

- 我接着查看了哪些源码文件
- 哪些细节是 graphify 看不到、但源码里能确认的

## 典型输出风格

### 示例 1：架构问题

- 先用 `graphify query "场景生成流程怎么工作"` 缩小范围
- 如果结果指向 `runner -> renderer -> evaluator`，先总结主链路
- 再只读取这几个节点对应文件，而不是全局乱看

### 示例 2：依赖关系问题

- 先用 `graphify path "config" "render"`
- 输出配置如何流入渲染流程
- 如果路径经过中间转换层，再补充解释关键节点

### 示例 3：影响分析问题

- 先用 `graphify affected "PromptBuilder"`
- 列出直接和间接受影响节点
- 再提示用户哪些测试或验证最值得先跑

## 质量标准

一个好的 graphify 驱动回答，应该满足：

- **先给结构，再给细节**
- **先缩范围，再读代码**
- **尽量少读无关文件**
- **能指出关键节点、路径、影响面**
- **修改后记得更新图谱，但只在需要时更新**

## 成功标准

当用户问代码库问题时，你的行为应表现为：

- 优先使用 graphify 的 `query / path / explain / affected`
- 不直接陷入大范围源码搜索
- 回答里能明确说出关键节点或关键路径
- 在代码修改后，**按需**执行 `graphify update .`
- 对高价值结论考虑使用 `graphify save-result`
